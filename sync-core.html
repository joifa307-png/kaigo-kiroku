5) 印刷テスト（iPad 切り分け）

LINE WORKS などアプリ内ブラウザでは、テスト1でオーバーレイの文言だけ出て、OSの印刷ダイアログが開かないことがあります。 これは多くの場合 アプリ側の WebView が window.print() を無効にしているためで、利用者設定だけでは直りません。 同じURLを Safari（または Chrome）で開き直して印刷テストしてください（LINE WORKSの「ブラウザで開く」等があればそれを利用）。

本アプリ（index.html）では印刷できないが、他ページではできる／プリンターは反応するがデータが出ない、といった場合に、 この Sync Core 単体で window.print() が通るかを確認するための最小テストです。

テスト1は「常に安定する基準」とは限りません。 特に テスト2（とくに HTML table）が NG になった直後にテスト1も NG になることはよくある挙動で、 ブラウザや iPad の印刷パイプ、AirPrint、プリンター側のジョブキューや内部状態が一度詰まったままになっている可能性が高いです（ページのバグというより、失敗した印刷ジョブのあとに何かが残るイメージ）。 そのため「2だけが悪い」と決めつけず、詰まりを解いてからもう一度テスト1だけ試す、という順序が切り分けに向きます。

切り分けの読み方（目安）
・テスト1がOK・テスト2（table）だけ繰り返し NGで、かつ table のあとに必ず詰まる → <table> 経由のPDF生成やプリンター側の解釈が弱い線。
・詰まったあとは何を印刷してもダメ → まず 1〜2分待つ／印刷ダイアログを確実に閉じる／プリンター電源再投入／必要なら Safari のタブを閉じて開き直しなどでキューを空にしてから再試験。

既定のテスト2は div＋グリッドの表風（table を使わない）です。下の「HTML table」は、table が原因か・詰まりかを分けるための比較用です。連打は避け、1回印刷するたびにプレビューで中身を確認してから次へ進むと状態が追いやすいです。

印刷ダイアログを閉じるとオーバーレイは自動で消えます。 「詰まった」場合はアプリ側ではキューを掃除できないため、上記の待ち・プリンター再起動を挟んでから、テスト1のみで復帰したか確認してください。
